== Performance and energy efficiency

Devices live in multiple layers

. sensor/actuator layer
. cloud/processor

Sending data to the cloud and getting it back in a reasonable time frame is hard

Cloud services could be thousands of km away

* high latency

Can cause problems in fields like health

Routing and server issues across the internet can also cause reliability problems

We can use the sensor itself to process the data

* provided it is powerful enough
** will never be more powerful than a remote server
* local actuator implements sensor results

Multiple of the same device can come to different conclusions depending on where/how they're placed

=== Cloud, edge and fog

Cloud computing refers to servers in a data center

* few (relatively)
* powerful
* well protected

Fog network is a set of low power servers

* a smartphone could be a fog server
* offer part of cloud server services
** evaluation
** formatting
** expanding/decoding
* placed close to the access point
* very low latency

Edge computing refers to smaller devices

* many
* smartphones
* laptops

== Interoperability and standards

Making vertical silos is useless (but it will make you a lot of money)

* your solutions will only solve your problems on your devices with your data
* you cannot apply them to other contexts
* device A from manufacturer A cannot communicated with device B from manufacturer B

Proprietary solutions require *secrets*

* security threat
* fewer eyes
* bugs can be problematic

=== Why standards?

Advantage for manufacturers and developers/researchers

Manufacturers develop solutions and sell solutions at high prices

Subsequent manufacturers will attempt to copy/reverse engineer/whatever

* this will cost you in the long run
* maintaining underlying techonlogies is expensive

Competing manufacturers often congregate in consortiums to maintain/develop underlying tech

* they compete with each other at the service level
* e.g. HDMI consortium, USB consortium

Too many standards cause fragmentation

* new silos are formed

=== Gateways

Translate protocols at the application level instead of translating low level protocols

* map application level behaviors into other application level behaviors
** or all behaviors above network

==== Service gateways

Act as mediators between different devices

* different vendors with same protocol
* brings IoT data flow to internet

==== Integration Gateaways

Implements a behavior map

* translates protocols
** n^2 order of translations
* allows for interoperability between devices from different manufacturers

Products like Alexa and Google Home

* they force other manufacturers to use their proprietary protocols

Big corporations have the ability to enforce integration gateways

Centralized IGs create single points of failure

* decentralize them
* parallelize them
* n order of translations

== IoT and other tech

=== Blockchain

Cooperative distributed system

Implements a ledger (i.e. a database)

* can only append
* cannot change previously entered data
** unless update is approved by all members
* permanent registry

Blockchain implementations can be used to run scripts

* transactions are stored in ledger

Can be used to monitor supply chains

* multiple independent actors work on supply chains
* e.g. drug company makes medicine that has to be stored in a specific condition
* if the drug arrives at the pharmacy improperly we can't know for sure where the container broke
* IoT devices can monitor the state of the package
* the state is certified by sensor data and transactions (e.g. shipping company -> final delivery) are recorded on the ledger

=== Federated learning

Don't have to send raw sensor data which could entail confidentiality/privacy issues

Send local model to orchestrator

* knowledge is combined
* new models are redeployed to all members of the federation

However this is weak to model poisoning

* device learns improperly
* bad model is sent to orchestrator
* bad knowledge is integrated
* bad knowledge is deployed to other devices

== Security in IoT

IoT conditions are relatively extreme

* low power
* potentially dangerous locations
* highly constrained
** low processing power
** low memory

Connecting these devices to the internet allows them to access cloud services

* higher power
* higher storage

Connecting them to the internet also opens them to security vulnerabilities

Manufacturers want to reach the market as fast as possible

* security is sidelined

The largest botnets in the world are powered by unsecured wifi cameras

=== Patching

If a vulnerability is found it should be patched

* very hard to do
** does the user do it?
** does the manufacturer do it?
** does the device even support updates?
*** firmware updates are sketchy

Bad for sensors

* can steal data

Really bad for actuators

* they can be made to misbehave
** imagine a wifi enabled pacemaker

=== IoT and Inference

IoT data can be abused to infer new information

* e.g. Strava maps were used to identify military bases and such

=== IoT Vulnerabilities

Many types

These are very common:

. weak hardcoded passwords
* especially if baked into firmware
. insecure networks
* no VLANs firewalls
. no updates
. bad update mechanisms

etc.

=== IoT security/privacy requirements

IUT-T (association) recommends certain practices for security

. communication and data management security
. security management
* no one size fits all
** scale down towards low end devices
. Mutual authentication and authorization
* devices should be authenticated with some method
* they should also be authorized to perform actions
. security audits

=== IoT Gateway security functions

If the physical location is secure then the gateway is the single point of entry

* otherwise you have more problems

Gateways should:

* perform authentication with devices
* ensure data security during transport/storage
* deliver updates
** firmware/software
